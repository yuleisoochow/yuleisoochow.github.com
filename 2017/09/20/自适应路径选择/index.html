Adaptive path selection

Design overview

目前市场上的许多扩展器都是双频双向并发的，并且在两个频带上连接到主AP（这里称为中央AP或CAP）。 如下图所示，可以将这样的网络视为在每个频带上具有回程接口，并且在每个频带上提供与附近STA的连接的AP接口（称为接入接口）。

传统上，这些DBDC扩展器实现了在向上游发送流量时使用的接口的固定选择。这通常是为了避免网络中可能导致广播风暴的桥接环路。这样的固定转发方案也可以与纯粹作为学习桥梁的CAP一起工作，因为CAP将在单个接口上学习RE（无论是有线还是无线）之后的RE和客户端的MAC地址。这意味着上行链路和下行链路路径是相同的，并且由RE如何选择上行链路路径来控制。

通常使用两个固定转发模型，即将流量转发到与与RE相关联的STA所使用的相同信道上的回程接口或将转发流量转发到另一个频带上的接口。这些方案中的每一个都有优缺点。交叉频段方法消除了访问链路和回程链路之间的自我争用的影响，有时可以导致更好的性能。然而，由于2.4 GHz和5 GHz（例如，通常为20 MHz和80 MHz）的带宽不同，这种交叉频带转发可能导致2.4 GHz回程接口成为5 GHz相关客户的瓶颈。另一方面，相同的频带桥接可能导致另一频带的利用不足。根据客户端在整个频段的分布情况，这可能会导致所有客户端的性能显着降低。

自适应路径选择（APS）允许在CAP和RE两者都实现该算法时，每个目的地或甚至每个流中采用不同的转发路径（以下有一些限制）。该算法有三个核心方面，即循环避免，拓扑和链路度量收集，以及实际的路径选择和负载平衡。这些将依次被考虑。

Loop avoidance(循环回避)

如上所述，当多个接口连接到同一个上游设备时，重要的是确保数据包不会以无限循环转发。可以使用传统的生成树协议来避免桥接拓扑中的循环，但是它们导致在主要端口（称为根端口）失败之前不使用上游面向接口之一。因为无线网络上的条件不像有线网络那样可预测，所以仅限于单个接口可能是次优的。

APS通过引入上游和下游端口的原理解决了这个问题，在实现中被称为非中继和中继端口。对于面向上游的端口，它们之间不允许桥接以避免环路。另外，只允许一个上行端口在下游端口之间传输广播流量，以避免看到广播帧两次。另一方面，对下游的港口也没有限制。帧之间可以转发帧。

确定端口是否必须是下游或上游端口当前基于启发式。 STA接口总是被认为是上游的，因此这些接口被放置在非中继组中。 AP接口和以太网接口被认为是下游接口，因此这些接口放置在中继组中。这可以处理RE作为无线扩展器的拓扑。当RE设备通过以太网或通过单独的PLC网络连接到CAP时，这些启发式将导致桥接回路。但是，如果此功能与RE布局和自动配置功能一起使用，则可以支持这种拓扑（只要设备使用以太网连接启动），因为STA接口被禁用。

Topology and link metric collection(拓扑和链接度量收集)

为了使路径选择算法做出关于如何转发流量的决策，需要收集网络拓扑信息和网络链路强度。通过使用IEEE1905.1拓扑发现，查询，响应和通知消息以及一些供应商特定的扩展来了解网络拓扑。发现消息使得节点能够确定哪些接口连接到其他IEEE1905.1设备，而查询和响应消息允许节点从远程节点获取完整的拓扑信息。该信息包括远程接口（该节点未直接连接的接口）和每个接口上学习的MAC地址。

除IEEE1905.1提供的信息外，还提供了附加信息，其中包括相关STA的列表（以帮助消除网络中给定的MAC地址实际位于何处）以及其他AP接口的Wi-Fi功能信息。当STA在本地关联以帮助其他节点快速更新其桥接信息时，节点还包括拓扑通知消息中的附加信息，因为正常拓扑通知不包括任何信息，并要求节点查询发送节点以确定实际改变的内容。

链路度量信息是进行智能转发决策的另一个关键因素。 IEEE1905.1定义了一个节点可以向另一个节点查询关于其链路的物理和MAC层特性的信息的方法。然而，由于标准没有明确定义如何衡量或解释这些度量标准，所以在链路指标查询/响应消息中包含额外的供应商特定指标。这些供应商特定的指标由发送节点估计的TCP和UDP流量的容量值组成。这些值用于确定给定客户端的路径。

此外，该标准仅定义了IEEE1905.1设备之间的链路度量。由于APS基于关于最终跳跃的信息（对于相关联的STA）进行决策，所以附加链路度量信息由具有该APS特征的节点交换。这些节点还周期性地将伪帧发送给彼此，以确保它们可以从Wi-Fi驱动器和固件获得更新的速率值，因为这在下面描述的容量计算中使用。

对于每种类型的流量（TCP和UDP），节点估计两个容量值。 UDP满容量是假设完全空闲介质可实现的应用层数据速率的估计。该值来自于速率控制算法选择的最后一个物理层数据速率。将最后的传输速率与两个阈值进行比较，以确定其落入的三个区域中的哪一个。应用不同的缩放因子（基于给定物理层速率的有效数据速率的模拟）来获得实际的满容量值。应用进一步的缩放因子来获得TCP的全容量值。该缩放因子估计使用TCP时所产生的附加协议开销的影响。然后，根据较低层报告的分组错误率进一步调整这些满容量值。

在获得满容量值之后，可用容量通过将全部容量缩放到介质上可用的剩余时间（达到可配置的最大利用率小于100％）来确定。这是在路径选择决策中使用的这个值。

Path selection and load balancing(路径选择和负载均衡)

自适应路径选择通过组合拓扑和链路度量信息来达到要为特定目的地或流量采用的最佳路径。在高层次上，它包含两个进程，一个用于选择网络中的默认路径（适应加载条件更改），另一个在过载期间更改网络中现有流的路径。

默认路径选择过程在每个节点上运行，并确定连接到网络中其他节点的所有MAC地址的最佳出口接口。这种确定是在周期性的基础上以及发生重大事件时进行的。这样的重要事件可以包括网络中的其他节点的拓扑变化和中等利用率的变化或物理层速率。

当确定给定MAC地址的默认出口接口时，会为每个可能的路径计算端对端路径。当目标MAC地址不是Wi-Fi STA（例如，它是一个AP设备本身或通过以太网连接到这种设备的设备）时，APS选择具有最佳可用容量的接口。

当目的地MAC地址是与相邻AP相关联的STA时，估计到达STA的所有可能的回程路径的端到端数据速率。当这条路径涉及对回程链路和接入链路使用相同的信道时，通过将两个可用容量相乘，然后将其除以它们的总和来估计自我争用的影响。当路径涉及用于回程和接入链路的不同信道时，将可用容量的最小值用作端到端路径。

例如，在下图所示的拓扑中，在回程和接入链路上使用2.4 GHz的路径的端对端数据速率将被估计为（Rb，2×Ra）/（Rb，2 + Ra），而在回程链路上使用5GHz的路径和接入链路上的2.4GHz的路径将被估计为min（Rb，5，Ra）。

另一个特殊情况是确定从一个RE（RE1）到另一个RE（RE2）或与RE2相关联的设备的路径。由于CAP将执行BSS内桥接，因此APS将考虑在每个频道上共享一个从RE1到RE2的虚拟链路。以上使用的自我竞争计算将应用于此虚拟链接。任何进一步的争用影响将适用于与RE2相关联的Wi-Fi STA的最终跳转。

自适应路径选择中的第二个过程是可以将活动流移动到新接口的负载平衡过程。流量由其目的地MAC地址和数据包中的某些字段（通常是源地址和目标地址和端口号）的8位散列来标识。由于哈希的大小有限，两个不同的流可能映射到相同的哈希。但是请注意，如果它们注定为两个不同的MAC地址，则它们仍将被视为单独的流。

这种流动移动过程可以通过网络拓扑的变化以及信道上的媒体利用率高于可配置值（默认为70％）来触发。对于拓扑变化，该算法根据可用容量寻找新的接口。响应中等订阅是一个多步骤的过程。首先，超额认购界面上的所有流量都会根据其对中等利用率的贡献进行排名。然后，从最高贡献者开始，检查具有到目的地的链接的每个接口。如果找到一个具有可配置的量（默认值为10％）的接口比流量更多的容量，则选择该接口。如果没有找到接口，则检查下一个最高流。这个过程一直持续到一个流动被移动或一定数量的流量被检查，没有一个候选人。

在流量由于超额预订而移动之后，任何进一步的负载平衡被延迟，直到更新的介质利用信息可用。这种行为阻止了太多的流量被移动，因为这种流量对利用率的影响在这一点上没有得到充分的反映。

NOTE 流的路径切换在本地通过流经的AP进行。 对于实际的最终客户端是透明的。

NOTE 虽然采用了自适应路径选择（APS）决策，但不可能适应高于目标频带的安全阈值的利用率。 TCP不能看到这个问题，只有当UDP流量的带宽超过90 Mbps（当2 GHz频段被超额订购）时才会注意到。 因此，如果必须将其移动到不同的接口，则需要重新启动流。

Implementation details(实施细节)

自适应路径选择的实现被分为两个组件 - 一个组件驻留在控制路径中，而另一个组件位于数据路径中。

Control Path – hyd

控制路径部分负责根据拓扑和链路度量变化配置桥接和选择路径。 hyd守护进程包含控制路径实现。 实现路径选择的关键模块如下：

  • tdService - 该模块负责基于IEEE1905.1消息传递学习网络拓扑。
  • pcwService - 该模块处理所有Wi-Fi接口的驱动程序的链接度量信息。 它将原始物理层指标转换为容量值，并将其提供给其他模块。 它还具有确保在APS负载平衡操作发生后更新中等利用率信息的逻辑。
  • psService - 这是实现上述路径选择算法的核心模块。 它对来自其他模块的事件做出反应，指示拓扑或链接度量的变化，并相应地重新编程桥接表。
  • heService - 此模块收集网络中活动流的周期性字节数样本，并估计其速率。 该信息由psService使用，同时确定流必须从一个接口移动到另一个接口的方式。

Data Path – Bridging infrastructure

自适应路径选择数据路径使用连接到桥接器的Linux内核Net过滤器钩子进行某些操作（例如转发和本地输入）来实现。 这些钩子在数据路径中实现APS的三个关键方面：

  • Preventing bridging loops(防止桥接循环) - 转发和泛洪钩可防止在上游（非中继）接口上收到的数据包在另一个上游接口上转发。它们还确保只有一个广播帧被传递到下游（中继）接口。
  • Default path selection control(默认路径选择控制) - 名称为H-Default的桥接表可以由用户空间来编程，以覆盖使用桥本身的转发数据库（FDB）所发生的桥接。该表由目的MAC地址和用于UDP和非UDP流量的接口的名称组成。
  • Active path selection control(主动路径选择控制) - 名为H-Active的桥接表可以控制用于单个流的接口。当H-默认或FDB中的条目匹配时，将创建此表中的条目。该条目跟踪最终目的地址，流的哈希，流的统计信息和出口接口（以及此处未涵盖的一些其他信息）。用户空间可以写入一个条目，以强制该流程使用不同的出口接口。

当水连接到桥上时，这些钩子被激活。当活动时，为给定的目的地址（DA）选择出口接口的过程如下：

  • 根据数据包中的字段计算流的散列，并在H-Active中查找以确定是否存在（DA，散列）对。如果是这样，更新统计信息并使用选择的出口接口。否则，继续下一步。
  • 在H-Default表中查找DA。如果发现匹配，则使用根据该数据包是UDP还是非UDP指定的出口接口。
  • 当FDB中有条目时，使用正常的基于FDB的转发规则。如果没有找到条目，则在所有下行（中继）端口和一个指定的广播上行（非中继）端口上，不包括接收到该数据包的接口的数据包。该逻辑也适用于广播/组播流量。
